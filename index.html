Eureka简介

Posted on 2020-05-25

Eureka基本作用

  1. 服务注册与发现
  2. 心跳与故障

Eureka 对服务注册

  1. 服务启动时，会生成一个服务注册对象registerInfo，向eureka（服务治理中心）注册
  2. 服务注册完成后，会从eureka拉取所有服务信息，缓存在本地，并每隔30s从eureka重新拉取服务信息
  3. 服务每个30s（可配置）向Eureka发送心跳信息。
  4. Eureka如果90s没有收到服务的心跳包，则认为服务宕机
  5. 服务停止时，会向eureka发送服务器停止信息
  6. Eureka每隔60s（定时任务，可配置）删除90s没有收到心跳发送过服务停止信息的服务
  7. Eureka接收到的心跳包不足正常的85%（可配置）则会进入保护模式，不删除服务信息
  8. 服务过期时间与服务刷新时间配置

Eureka 自身管理

  1. eureka使用concurrentHashMap来管理服务注册表
  2. 服务发现不是直接从服务注册表中获取信息，而是从缓存中获取信息，目的是支持高并发
  3. eureka缓存有两层
    1. readWriteMap
    2. readOnlyMap
  4. readWriteMap 直接从服务注册表中获取数据，与服务注册表保持一致
    1. 当有服务来注册或服务被删除当时候，会主动让readWriteMap中的数据过期
  5. readOnlyMap通过后台线程定时（30s可配置）从readWriteMap中同步信息，非实时。若readOnlyMap中的数据为空，则会从readWriteMap中获取。
  6. eureka中有定时任务，定时检查服务注册表中的过期信息并清理，同时使redisWriteMap中的信息失效。

eureka集群架构原理

当eureka集群部署时，服务提供者向eureka注册了服务，该eureka则会调用其他eureka的服务注册功能。

分布式事务简介

在多个数据库服务之间实现ACID

2PC 两阶段提交

引入一个事务管理器作为事务的协调者

  1. 预提交
  2. 实际提交

在程序执行复杂数据操作时，开启事务，sql执行完成后不commit，将执行结果告知事务管理器，等得到了该事务中所有sql语句的执行结果后，再集体通知所有数据库进行commit或者rollback。

并且，分布式事务不能百分百解决，只能提高成功概率，没办法解决的只能人工介入。

一般2pc使用是一个应用层服务管理两个及以上数据库服务

TCC 两阶段补偿型方案 （Try-Comfirm-Cancel）

  1. 预留资源
  2. 确认资源

每个服务都要提供两个接口，一个用于第一阶段的资源预留行为，另一个用于确认、回滚

如何预留资源

数据库表本来只有一个库存数字段，TCC方案后，再加上一个冻结库存数字段。

第一阶段，购买商品，是先将冻结库存数字段+1，第二阶段，才是对库存数字段和冻结库存数字段一起操作。

  1. 2pc粒度比较大，所有数据库均被应用持有，直到事务完成，性能较低

命名的意义

Posted on 2019-11-25 Edited on 2020-05-22

前提

在我们讨论命名的时候，我们需要列出以下前提

  1. 被命名的代码是需要共同合作的
  2. 被命名的代码是需要多次开发，进入版本迭代的

若代码满足以上两者任意一条，则需要对命名进行规范

如何执行

名副其实

变量

对于某个变量而言，名称应正确的表示该变量的含义
例如：

  • 主键到实体的Map： id2VOMap

方法

对于方法而言，名称应正确的表示该方法所实现的事情
例如：

  • 判断是否存在: isExist
  • 通过id获取实体列表：getVOListById

魔法值

对于魔法值而言，应该放入全局静态变量中，标识其含义并给出示意
例如：

  • redis的过期时间： REDIS_EXPIRATE_TIME

需要注意的是，对于值相同，但是含义不同的魔法值，应该要建立多条变量

命名参考链接

  • codelf

根据上下文设置名称

可以根据当前操作的上下文来设置一个变量的名称，例如：

  • 对于更新数据而言，一个可以是source，一个是target

名称的长度

这是一个取舍问题，在尽量表达清楚的同时，名称需要尽量的短。若名称本身已经影响到了对代码的阅读，那么更好的方案应该是缩减命名并添加备注。

总接

一切的命名，都是为了更加容易理解而设置的，命名应该是有更好的名称，则马上修改。这个事情本身没有尽善尽美之说，只能在能力范围内尽量通俗易懂的命名。

git 合作开发的一种流程

分支分类

  • master
  • dev
  • alpha
  • Regression （没有使用过，预想）
  • bug
  • tag（发布时使用）

测试状态分类

  1. 开发中
  2. 自测完成
  3. 测试人员测试完成
  4. 回归测试完成
  5. 上线

master与tag

  • master为主干分支，是分支中最重要的一个分支，所有回归测试完成准备上线的分支均可以合并到master分支
  • tag称为版本，是用于发布正式环境的一种功能， 不参与分支之间交互。当本版本所有功能均合并到master，准备发布的时候，打出一个tag，标示版本号，并使用tag代码作为发布的代码

dev

develop分支为功能分支，dev分支的数量与功能的数量相同，其划分规则不一，可能根据模块划分，可能根据业务边界划分，该分支主要负责对象为开发人员

alpha

alpha分支为测试分支，所有需要进行功能测试的代码均可以合并到alpha分支上，需要注意的是，测试分支不能合并到任何分支，只能由其他分支合并到测试分支，并且测试分支应该是可以被随时删除的，由功能测试人员来保证测试分支的存在性质。

regression

regression分支为回归分支，该分支的作用是在功能测试通过后，从master切出一个版本的回归分支，然后将测试通过的功能分支合并到该分支上，再次发布到测试环境，让测试进行回归测试

bug

bug分支为修复线上问题的分支，直接从master切出，并进行微小的改动，改动过后，直接作为回归分支由测试进行测试通过后，直接合并到master

总结

从master出发，开发功能时，切出一个dev_function分支，自测完成后，合并到alpha分支发布测试让测试人员测试，测试人员测试通过后，从master切出regression分支，并合并dev_function分支发布到测试环境让测试人员进行回归测试，通过后则将regression分支合并到master，从master打一个tag分支，发布到正式环境。

git cherry-pick的一种使用姿势

git cherry-pick 功能的作用是遴选，从某个分支中挑选需求的改动到开发分支中，目前该功能在正常的合作流出应该不会使用，一般使用该功能的时候，说明分支出现了人为的失误，或者管理出现了失误。

情景一：功能改动过时

由人员A在改动尺码管理的功能，创建了一系列的表，在表单中添加了大段的业务逻辑去兼容原来的尺码，人员A做了如下事项

  1. 创建分支 dev_form_size_table_rebuild
  2. 创建表，并将sql操作自动生成代码后提交
  3. 修改业务逻辑，去使用新的数据源，并兼容原有数据源
  4. 其分支内容大概可以分成如下情况，从上倒下时间是从前到后
    1. sql操作提交
    2. 使用sql操作，创建新的尺码数据来源
    3. 在表单中使用新的尺码数据来源，并兼容原有尺码数据
      但是在改造完成后，人员A离职，并且表单的业务逻辑进行了较大的变动，那么人员A在其分支中第三点（在表单中使用新的尺码数据来源，并兼容原有尺码数据）改动已经没有必要，那么需要进行如下操作
  5. 表单改动分支为：dev_form_rebuild
  6. 使用git cherry-pic从dev_form_size_table_rebuild中挑选出 1 与 2 的改动
  7. 在表单改动中使用新的尺码数据来源，并兼容原有尺码数据
    至此，使用git cherry-pick 功能将离职人员撰写有用的代码剥离出来，直接舍弃了其分支，以免影响表单的业务逻辑改造

当合并错误分支后，该如何处理

参与的分支

  1. master
  2. dev_function
  3. dev_group
  4. alpha

    发生的事情

    当两个开发人员分别开发dev_function与dev_group时，共同合并到了alpha，将alpha作为他们俩的功能测试分支，此时，其中一个开发人员发生了错误，将alpha分支合并到了dev_function分支上，那么，目前的分支图如下所示

解救方式

此时需要做的事情是，寻找到dev_function上最后一次独立提交的分支，并将此分支的版本号作为分支切出，并从这个分支中切出dev_function分支，并将原来的dev_function分支删除，强制更新。

还有一种情况，那就是将alpha分支合并到dev_function分支后，开发人员没有发现，继续在上面做提交操作该怎么办？

  1. 继续按照上述方法寻找到合并alpha之前，dev_function上最后一次独立提交的分支，复制其版本号作为分支，并从这个分支上切出dev_function_new
  2. 通过cherry-pick的方式，将开发人员在dev_function分支上合并了alpha以后的所有操作均遴选出来，按照时间的先后顺序合并到dev_function_new分支上，解决冲突并提交，自此完成了将dev_group分支上功能从dev_function_new分支上移除的操作

    总结

    最重要的就是寻找分支中最后一次独立提交的版本号，git会备份该版本提交时所有的代码情况

对于merge与rebase使用场景探讨

区别

merge与rebase均用来合并分支，两者的区别在于

  1. merge会保留合并分支之间的关系，时间轴，只需要解决一次冲突
  2. rebase不会保留分支之间的关系与时间轴，需要解决多次冲突

    merge 图

    rebase 图

    使用场景

  3. rebase绝对不能在公共分支中使用
  4. rebase一般使用情况是在自己开发的分支不提交的情况下使用，直接rebase到自己在远程的开发分支上，减少不必要分支的关系
    ps：不探讨merge与rebase的操作手册

merge request的作用

merge request主要作用是代码审核，一般是由两个人员之间互相审核，检查代码规范以及合并内容是否在开发范围之内，避免一下情况

  1. 不符合代码规范
  2. 合并内容不在范围内，例如合并了别人的功能

总结

git作为版本控制工具，其功能是非常多的，但是在操作过程中需要注意很多事情，在多人合作使用的情况下，需要制定规范来约束其使用场景。

本文中讨论的所有内容均基于多人合作的场景。

“mybatis返回的自增主键”

前言

本文仅仅记录返回主键的结论，不进行源码分析

结论

以下insertOrUpdate 为insert on duplicate key update

  • 使用@Param注入入参时，在任何情况下keyProperty必须用”@Param的入参名字.主键属性名” (entity.id)的形式才能正确解析到入参对象上并设置主键
  • 批量insert时，采用@Param的注解是没有办法返回主键的，所以paramenterType在返回主键上比@Param更具有通用性。
  • insertOrUpdate时，selectKey只能正确返回插入时候的主键，无法正确返回更新时候的主键，useGenerateKeys在插入或者更新的时候都能够正确的返回主键。所以在insertOrUpdate时采用useGeneratedKey。
  • 批量insertOrUpdate时候，任何情况都不能正确的返回主键，所以程序逻辑不能依赖返回的主键。
