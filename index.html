Zookeeper集群特性

Posted on 2020-05-25

zookeeper集群

目的是为了保证系统性能，能承载更多的客户端连接数专门提供的机制。通过集群可以实现以下功能

  1. 读写分离：提高承载，为更多客户端提供连接，并保障性能。
  2. 主从自动切换：提高服务容错性，部分节点故障不会影响整个服务集群。

半数以上运行机制说明

集群至少有三个服务，并强烈建议使用奇数个服务器。因为zookeeper通过判断（n/2 并四色五入）个节点是否存活判断整个服务是否可用。例如3个节点，2个节点宕机表明整个集群不可用，而用4个节点，也是两个节点宕机表明整个集群不可用。

集群部署

  1. 节点Id：服务id手动指定1至125之间的数字，并写到对应服务节点的{dataDri}/myid文件中。
  2. IP地址：节点的远程IP地址。（可以相同，但是不建议，相同无法达到容错的目的，称之为伪集群）。
  3. 数据同步端口：主从同步数据复制端口（伪集群中端口不能重复）
  4. 选举断口：主从节点选举端口（伪集群中不能重复）

集群角色说明

一共有三种角色

  1. leader
    • 主节点，又称领导节点。用于写入数据，通过选举产生，如果主节点宕机将会选举新的主节点。
  2. follower
    • 子节点，又称追随者。用于实现数据的读取。同时他也是主节点的备选节点，并拥有投票权。
  3. observer
    • 次级节点，又称观察者。用于读取数据，与follower区别在于没有投票权，不能选举为主节点，并在计算集群可用状态的时候不会将observer计算在内
    • 配置示例：server.4=127.0.0.1:2889:3889:observer

选举机制

通过status可以查看节点角色等状态。

投票机制说明

  1. 所有节点的事务id一样的情况下
    1. 第一轮投票给自己
    2. 第二轮投票给myid比自己大的相邻节点
    3. 如果得票超过半数，选举结束。
  2. 事务id不一样的情况下
    1. 最新并且得票最多的节点选举为leader

选举触发

两种情况下会触发选举

  1. 服务节点初始化启动
  2. 半数以上节点无法和Leader建立连接（Leader宕机）

当follower节点初始启动时候会在集群中寻找Leader，如果找到则与Leader建立连接，自身状态变为Follower。如果没有找到Leader，当前节点将变化LOOKING，进入选举流程。

在集群运行期间，如果有follower节点宕机，只要不超过半数并不会影响整个集群服务的正常运行。但是如果Leader宕机，将暂停对外服务，所有Follower进入LOCKING状态，进入选举流程。

数据同步机制

zookeeper的数据同步是为了保证各个节点中数据的一致性，同步时涉及两个流程

  1. 正常的客户端数据提交
  2. 集群某个节点宕机再恢复后的数据同步

客户端写入流程说明

  1. client向zk的server发送写请求，如果该server不是leader，则会将该写请求转发给Leader server，leader将请求的事务以proposal形式分发给follower；
  2. 当follower收到leader的proposal时，根据接收的先后顺序处理proposal；
  3. 当Leader收到follower针对某个proposal过半的ack后，则发起事务提交，重新发起一个commit的proposal
  4. Folllower收到commit的proposal后，记录事务提交，并把数据更新到内存数据库；
  5. 当写成功后，反馈给client

服务节点初始化同步

在集群运行过程中如果有一个follower宕机了，由于宕机节点数没有过半，集群仍然正常服务。当leader收到新的客户端请求，此时无法同步给宕机的节点，造成数据不一致。为了解决这个问题，当节点启动时，首先找当前的leader比对数据是否一致，不一致则开始同步，同步完成后再对外提供服务。

对比数据版本是否一致是通过ZXID事务ID来确定的，比Leader小就需要同步。

ZXID说明

ZXID是一个长度为64位的数字，其中低32位是递增的（加1），任何数据变动均会使低32位递增。高32位是Leader的周期编号，每当选举一个新的Leader时，Leader就从本地事务日志中取出ZXID，然后解析出高32位，进行加1，再将低32位全部设置为0。这样就保证了每次新选举Leader后ZXID的唯一性与单调递增性。

类似于版本号的功能。

Eureka简介

Posted on 2020-05-25

Eureka基本作用

  1. 服务注册与发现
  2. 心跳与故障

Eureka 对服务注册

  1. 服务启动时，会生成一个服务注册对象registerInfo，向eureka（服务治理中心）注册
  2. 服务注册完成后，会从eureka拉取所有服务信息，缓存在本地，并每隔30s从eureka重新拉取服务信息
  3. 服务每个30s（可配置）向Eureka发送心跳信息。
  4. Eureka如果90s没有收到服务的心跳包，则认为服务宕机
  5. 服务停止时，会向eureka发送服务器停止信息
  6. Eureka每隔60s（定时任务，可配置）删除90s没有收到心跳发送过服务停止信息的服务
  7. Eureka接收到的心跳包不足正常的85%（可配置）则会进入保护模式，不删除服务信息
  8. 服务过期时间与服务刷新时间配置

Eureka 自身管理

  1. eureka使用concurrentHashMap来管理服务注册表
  2. 服务发现不是直接从服务注册表中获取信息，而是从缓存中获取信息，目的是支持高并发
  3. eureka缓存有两层
    1. readWriteMap
    2. readOnlyMap
  4. readWriteMap 直接从服务注册表中获取数据，与服务注册表保持一致
    1. 当有服务来注册或服务被删除当时候，会主动让readWriteMap中的数据过期
  5. readOnlyMap通过后台线程定时（30s可配置）从readWriteMap中同步信息，非实时。若readOnlyMap中的数据为空，则会从readWriteMap中获取。
  6. eureka中有定时任务，定时检查服务注册表中的过期信息并清理，同时使redisWriteMap中的信息失效。

eureka集群架构原理

当eureka集群部署时，服务提供者向eureka注册了服务，该eureka则会调用其他eureka的服务注册功能。

分布式事务简介

在多个数据库服务之间实现ACID

2PC 两阶段提交

引入一个事务管理器作为事务的协调者

  1. 预提交
  2. 实际提交

在程序执行复杂数据操作时，开启事务，sql执行完成后不commit，将执行结果告知事务管理器，等得到了该事务中所有sql语句的执行结果后，再集体通知所有数据库进行commit或者rollback。

并且，分布式事务不能百分百解决，只能提高成功概率，没办法解决的只能人工介入。

一般2pc使用是一个应用层服务管理两个及以上数据库服务

TCC 两阶段补偿型方案 （Try-Comfirm-Cancel）

  1. 预留资源
  2. 确认资源

每个服务都要提供两个接口，一个用于第一阶段的资源预留行为，另一个用于确认、回滚

如何预留资源

数据库表本来只有一个库存数字段，TCC方案后，再加上一个冻结库存数字段。

第一阶段，购买商品，是先将冻结库存数字段+1，第二阶段，才是对库存数字段和冻结库存数字段一起操作。

与2pc的区别

  1. 2pc粒度比较大，所有数据库均被应用持有，直到事务完成，性能较低
  2. TCC每次数据库操作有，都是commit了，释放了锁，将锁的粒度限制在

分布式锁

Posted on 2020-05-21 Edited on 2020-05-25

分布式锁的特性

  1. 安全性：任意时刻，只有一个client能够获得锁（排他性）
  2. 避免死锁：client最终一定能获得所，即使锁住某个资源的he护短在释放锁之前崩溃或者网络不可达
  3. 容错性：锁服务集群大部分节点存活，则加锁服务可用

redis如何实现分布式锁

上锁

SET KEYID 随机值 NX PX 30000

  • KEYID：键值
  • 随机值：value。
  • NX：如果此时redis已经有了键值，则返回nil，如果没有这个值，则返回ok
  • PX：过期时间
  • 30000：过期时间长度 单位毫秒

LUA脚本版本

解锁

通过lua脚本删除

上锁的value使用随机值的原因

为了自己持有的锁不被别的线程删除，例如A服务上锁后，线程执行时间超过了过期时间，当线程结束后，如果服务B上了一样的锁，则服务A会将服务B的锁删除。所以在删除锁的时候，还需要对value进行校验

优劣

单机情况下

  1. redis宕机，则分布式锁机制失效
  2. redis如果是cluster，多master的情况下，则无法定位锁所在的位置

redLock算法

在redis cluster场景下，有5个redis master实例，执行如下步骤获取锁

  1. 获取当时时间，单位是毫秒
  2. 与普通情况类似，轮序尝试在每个master节点上创建锁，过期时间比较短，一般就几十毫秒
  3. 尝试在大多数节点上建立一个锁，n/2+1，5个节点在3个节点上创建锁
  4. 客户端计算建立好锁的时间，如果建立锁的时间小于超时时间，则创建成功
  5. 如果锁创建失败了，则依次删除这个锁
  6. 如果别人创建了一把分布式锁，你就得不断的轮询尝试创建获取锁

Zookeeper如何实现分布式锁

普通实现

  1. 尝试获取锁，创建临时节点
  2. 如果临时节点创建失败，则创建一个监听器
  3. 监听器回调以后，再次尝试创建临时节点

基于临时顺序节点实现

  1. 服务A1尝试获取锁a，并创建local/100000的锁，服务A2尝试获取锁a，并创建local/100001的锁，服务A3尝试获取锁a，并创建local/100002的锁
  2. 服务A创建锁以后，获取所有临时节点顺序锁，并进行排序，那么A1的锁在最前面，就可以获取到锁，A2的锁在第二个，他会向自己前面的临时节点（local/100000）注册一个监听器，A3相同，监听local/100001
  3. 当服务A1释放锁（删除临时节点local/100000）后，则zk会通知服务A1，服务A1直接获取锁；当服务A1释放锁（local/100001）后，服务A3会得到通知，并获取锁

redis与zookeeper实现的锁有哪些优劣

  1. reids锁实现比较麻烦，必须轮询上锁
  2. 将redis上锁的服务宕机了，则必须等待超时
  3. zk使用监听的模式回调
  4. zk使用心跳的模式检测临时节点是否有必要存在

分布式ID

Posted on 2020-05-21 Edited on 2020-05-25

分布式ID

为什么使用分布式ID

在业务量大的情况下，主从数据库没办法满足要求，需要进行数据库分库分表，那么就需要一份全局唯一的ID，称之为分布式ID

分布式ID需要满足的条件

全局唯一，高性能（不能成为系统瓶颈），高可用，好接入，趋势递增（非严格要求）

分布式ID的生成方式

  1. UUID
  2. 数据库自增ID
  3. 数据库多主模式
  4. 号段模式
  5. redis
  6. 雪花算法（snowflake）
  7. 滴滴出品（TinyID）
  8. 百度（uid-generator）
  9. 美团（Leaf）

UUID

  • 优点：生成简单，资源消耗小，唯一性
  • 缺点
    1. 无序字符串，不具备自增趋势
    2. 没有具体的业务含义
    3. 长度过长，对mysql性能效果较大，无序会导致数据位置频繁变动

数据库自增ID

基于数据库对auto_increment自增id，单独部署一个mysql实例用来生成ID

  • 优点：实现简单，数值类型查询速度块
  • 缺点：DB单点存在宕机风险，无法扛住高并发场景

数据库多主模式（集群模式）

多个数据库间，设置起始值与步长（每次新增大小）

  • 优点：解决单点数据库问题
  • 缺点：不利于后续扩容，并且单点数据库压力依旧很大，无法满足高并发场景

基于数据库号段

表设计

  • id
  • max_id 当前最大Id
  • step 步长
  • biz_type 业务类型
  • version 版本号 乐观锁，用于保证并发

号段用完了，就可以再次申请，update max_id = max_id + step 范围是 (max_id , max_id+step)。

redis

通过redis的原子操作 incr来实现自增，需注意redis持久化

雪花算法（snowflake）


使用Long 类型， 一个Long 类型占8个字节，每个字节有8bit，一共64 bit

组成

  1. 正整数 （1bit） 默认为0
  2. 时间戳（41bit）毫秒级，用当前时间减去固定开始时间 大概是用69年
  3. 工作机器Id（10bit）workId，可以灵活配置
  4. 序列号部分（12bit）自增值支持同一毫秒同一个节点可以生成4096个ID

百度（uid-generator）

雪花算法的变种，需要配合数据库使用，新增worker_node表，启动时从里面获取workerId

  1. workId 占用22bit
  2. 时间 占用28bit 秒级

美团（leaf）

同时支持号段模式和snowflake算法

滴滴（TinyId）

基于号段模式，更好接入，不需要重复造轮子

    2. CPU消耗方面，倒叙方式每次读写都需要额外调用一次reverse函数，而hash需要额外调用crc32()函数，仅从函数复杂度来说，是reverse函数的CPU消耗更小一点
    3. 查询效率上说，hash字段的查询性能相对更稳定一些。crc32算出的冲突值会比较小，可以认为每次查询平均扫描行数接近1.而倒叙索引还是使用前缀索引的方式，还是会增加扫描行数。

命名的意义

Posted on 2019-11-25 Edited on 2020-05-22

前提

在我们讨论命名的时候，我们需要列出以下前提

  1. 被命名的代码是需要共同合作的
  2. 被命名的代码是需要多次开发，进入版本迭代的

若代码满足以上两者任意一条，则需要对命名进行规范

如何执行

名副其实

变量

对于某个变量而言，名称应正确的表示该变量的含义
例如：

  • 主键到实体的Map： id2VOMap

方法

对于方法而言，名称应正确的表示该方法所实现的事情
例如：

  • 判断是否存在: isExist
  • 通过id获取实体列表：getVOListById

魔法值

对于魔法值而言，应该放入全局静态变量中，标识其含义并给出示意
例如：

  • redis的过期时间： REDIS_EXPIRATE_TIME

需要注意的是，对于值相同，但是含义不同的魔法值，应该要建立多条变量

  • codelf

根据上下文设置名称

可以根据当前操作的上下文来设置一个变量的名称，例如：

  • 对于更新数据而言，一个可以是source，一个是target

名称的长度

这是一个取舍问题，在尽量表达清楚的同时，名称需要尽量的短。若名称本身已经影响到了对代码的阅读，那么更好的方案应该是缩减命名并添加备注。

总接

一切的命名，都是为了更加容易理解而设置的，命名应该是有更好的名称，则马上修改。这个事情本身没有尽善尽美之说，只能在能力范围内尽量通俗易懂的命名。

git 合作开发的一种流程

分支分类

  • master
  • dev
  • alpha
  • Regression （没有使用过，预想）
  • bug
  • tag（发布时使用）

测试状态分类

  1. 开发中
  2. 自测完成
  3. 测试人员测试完成
  4. 回归测试完成
  5. 上线

master与tag

  • master为主干分支，是分支中最重要的一个分支，所有回归测试完成准备上线的分支均可以合并到master分支
  • tag称为版本，是用于发布正式环境的一种功能， 不参与分支之间交互。当本版本所有功能均合并到master，准备发布的时候，打出一个tag，标示版本号，并使用tag代码作为发布的代码

dev

develop分支为功能分支，dev分支的数量与功能的数量相同，其划分规则不一，可能根据模块划分，可能根据业务边界划分，该分支主要负责对象为开发人员

alpha

alpha分支为测试分支，所有需要进行功能测试的代码均可以合并到alpha分支上，需要注意的是，测试分支不能合并到任何分支，只能由其他分支合并到测试分支，并且测试分支应该是可以被随时删除的，由功能测试人员来保证测试分支的存在性质。

regression

regression分支为回归分支，该分支的作用是在功能测试通过后，从master切出一个版本的回归分支，然后将测试通过的功能分支合并到该分支上，再次发布到测试环境，让测试进行回归测试

bug

bug分支为修复线上问题的分支，直接从master切出，并进行微小的改动，改动过后，直接作为回归分支由测试进行测试通过后，直接合并到master

总结

从master出发，开发功能时，切出一个dev_function分支，自测完成后，合并到alpha分支发布测试让测试人员测试，测试人员测试通过后，从master切出regression分支，并合并dev_function分支发布到测试环境让测试人员进行回归测试，通过后则将regression分支合并到master，从master打一个tag分支，发布到正式环境。

git cherry-pick的一种使用姿势

git cherry-pick 功能的作用是遴选，从某个分支中挑选需求的改动到开发分支中，目前该功能在正常的合作流出应该不会使用，一般使用该功能的时候，说明分支出现了人为的失误，或者管理出现了失误。

情景一：功能改动过时

由人员A在改动尺码管理的功能，创建了一系列的表，在表单中添加了大段的业务逻辑去兼容原来的尺码，人员A做了如下事项

  1. 创建分支 dev_form_size_table_rebuild
  2. 创建表，并将sql操作自动生成代码后提交
  3. 修改业务逻辑，去使用新的数据源，并兼容原有数据源
  4. 其分支内容大概可以分成如下情况，从上倒下时间是从前到后
    1. sql操作提交
    2. 使用sql操作，创建新的尺码数据来源
    3. 在表单中使用新的尺码数据来源，并兼容原有尺码数据
      但是在改造完成后，人员A离职，并且表单的业务逻辑进行了较大的变动，那么人员A在其分支中第三点（在表单中使用新的尺码数据来源，并兼容原有尺码数据）改动已经没有必要，那么需要进行如下操作
  5. 表单改动分支为：dev_form_rebuild
  6. 使用git cherry-pic从dev_form_size_table_rebuild中挑选出 1 与 2 的改动
  7. 在表单改动中使用新的尺码数据来源，并兼容原有尺码数据
    至此，使用git cherry-pick 功能将离职人员撰写有用的代码剥离出来，直接舍弃了其分支，以免影响表单的业务逻辑改造

当合并错误分支后，该如何处理

参与的分支

  1. master
  2. dev_function
  3. dev_group
  4. alpha

    发生的事情

    当两个开发人员分别开发dev_function与dev_group时，共同合并到了alpha，将alpha作为他们俩的功能测试分支，此时，其中一个开发人员发生了错误，将alpha分支合并到了dev_function分支上，那么，目前的分支图如下所示

解救方式

此时需要做的事情是，寻找到dev_function上最后一次独立提交的分支，并将此分支的版本号作为分支切出，并从这个分支中切出dev_function分支，并将原来的dev_function分支删除，强制更新。

还有一种情况，那就是将alpha分支合并到dev_function分支后，开发人员没有发现，继续在上面做提交操作该怎么办？

  1. 继续按照上述方法寻找到合并alpha之前，dev_function上最后一次独立提交的分支，复制其版本号作为分支，并从这个分支上切出dev_function_new
  2. 通过cherry-pick的方式，将开发人员在dev_function分支上合并了alpha以后的所有操作均遴选出来，按照时间的先后顺序合并到dev_function_new分支上，解决冲突并提交，自此完成了将dev_group分支上功能从dev_function_new分支上移除的操作

    总结

    最重要的就是寻找分支中最后一次独立提交的版本号，git会备份该版本提交时所有的代码情况

对于merge与rebase使用场景探讨

区别

merge与rebase均用来合并分支，两者的区别在于

  1. merge会保留合并分支之间的关系，时间轴，只需要解决一次冲突
  2. rebase不会保留分支之间的关系与时间轴，需要解决多次冲突

  3. rebase绝对不能在公共分支中使用
  4. rebase一般使用情况是在自己开发的分支不提交的情况下使用，直接rebase到自己在远程的开发分支上，减少不必要分支的关系

merge request的作用

merge request主要作用是代码审核，一般是由两个人员之间互相审核，检查代码规范以及合并内容是否在开发范围之内，避免一下情况

  1. 不符合代码规范
  2. 合并内容不在范围内，例如合并了别人的功能

总结

git作为版本控制工具，其功能是非常多的，但是在操作过程中需要注意很多事情，在多人合作使用的情况下，需要制定规范来约束其使用场景。

本文中讨论的所有内容均基于多人合作的场景。
